## Context

仓库当前同时存在两套相互冲突的说法：

- 一套旧说法来自 `chat-compact` 相关 spec 与部分运行时实现，认为 `compact` 工具只有在上下文预算达到某个可见阈值后才会对模型暴露。
- 另一套更新后的产品意图认为，只要线程开启了 `auto_compact`，模型就应该立刻拥有 `compact` 这个能力；预算占比、容量提示和预警信息最多只能帮助模型判断“现在要不要调用”，而不应该决定“能不能调用”。

这次 change 的目标是先把规范收敛到后一套语义，避免后续实现继续被旧 spec 带偏。

## Goals / Non-Goals

**Goals:**

- 明确 `auto_compact` 与 `compact` 工具的正式契约：开启即暴露。
- 移除 canonical spec 中“预算达到可见阈值才暴露工具”的规范性描述。
- 明确预算阈值若继续保留，其职责只能是提示强弱、早期预警或 runtime compact 判定，而不是工具显隐。
- 为后续代码清理提供清晰边界，避免实现层继续同时兼容两套语义。

**Non-Goals:**

- 本次不直接修改运行时代码或配置字段。
- 本次不决定容量提示最终以什么消息形态注入。
- 本次不调整 runtime compact 的硬阈值触发机制。
- 本次不处理通过 command 中途切换 `auto_compact` 的能力。

## Decisions

### 1. `compact` 工具可见性以 `auto_compact` 开关为准，不再以预算阈值为准

新的规范要求是：

- `auto_compact = false` 时，模型看不到 `compact`
- `auto_compact = true` 时，模型立即看到 `compact`

预算占比不再参与工具显隐判定。

拒绝方案：

- 继续保留“达到可见阈值后才暴露”的设计  
  原因：这会让模型在最需要提前规划压缩时反而看不到工具，也和当前产品意图冲突。

### 2. 预算信息与容量提示从“工具显隐条件”降级为“辅助决策信息”

若系统继续输出容量百分比、早期预警文案或相关提示，它们只能承担两类职责：

- 帮助模型判断当前是否接近上下文上限
- 帮助模型判断是否应尽快调用 `compact`

它们不得再被描述成工具可见性的前置条件。

拒绝方案：

- 把预算提示和工具显隐继续绑在一起  
  原因：这会让两个独立概念再次耦合，后续实现和测试都更容易出现歧义。

### 3. 先通过 change 修改 canonical requirement，再在实现阶段清理代码与配置语义

这次 change 先解决“规范写错”的问题。实现层后续再按新的 canonical requirement 统一：

- AgentLoop 中基于预算判断 `compact_visible` 的逻辑
- 相关配置说明和字段命名
- 测试中把阈值当作显隐条件的断言
- 仍沿用旧术语的文档

这样可以避免在规范尚未统一前提前改代码，导致后续又被旧 spec 拉回去。

拒绝方案：

- 不起 spec change，直接改代码  
  原因：问题根源就在 spec，本次需要先把规范矫正。

## Risks / Trade-offs

- [Risk] 现有实现和测试仍按旧语义工作，短期内 spec 与代码会暂时不一致。  
  Mitigation: 在 tasks 中明确后续实现需要同步清理运行时逻辑、配置语义和测试。

- [Risk] 仓库里可能仍有其他活跃 change 或说明文档复述旧语义。  
  Mitigation: 后续实现阶段统一执行一次全文审计，把仍把阈值当作显隐条件的内容全部对齐。

- [Risk] `tool_visible_threshold_ratio` 这类历史字段在实现清理前仍会制造误导。  
  Mitigation: 先在 spec 中明确其不再控制工具显隐，后续实现时再决定是删除、重命名还是仅保留为提示强弱参数。
